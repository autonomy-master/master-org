## 2. 構成設計（構成作家）

執筆部の活動には最低限以下のレイヤーが必要:
1. **テーマ**: 何について書くか（個人開発の道のり / 技術解説 / ビジネスのリアル / etc）
2. **媒体**: どこに出すか（ブログ / Zenn / note / Substack / 自前サイト / 本）
3. **読者像**: 誰が読むか
4. **頻度・粒度**: 単発 / 連載 / 本（連載なら何本構想）
5. **目的**: 認知獲得 / SEO / コミュニティ形成 / 収益直結

提案: オーナーに **テーマ × 媒体 × 目的** の3軸で選んでもらう質問項目を作る。

## 3. 執筆（ライター）

執筆できる文体は多数あるが、オーナー本人の声と乖離した文体は読者に偽物感を与える。
> 「個人開発・SaaS」と聞くだけだと、技術ブログ系（Zenn）か、ビジネス系（note の有料）か、
> どちらでも書けるが文体が180度変わる。本人の好みを聞かないと外す。

提案: 過去発信があれば文体抽出。なければ最低でも「読みたい記事の例（誰々のあの記事）」を1〜3本もらう。

## 4. 批評ラウンド（読者代弁者）

【離脱リスク】
- 想定読者が分からないので「私は誰として読めばいいか」が決まらない。これは原稿を書く前の致命的な空白。

【刺さらない箇所】
- 仮想読者が「個人開発に興味のある人」程度の解像度では刺さる文章は書けない。
  実例: 「フリーランスの30代エンジニアで、自分の SaaS で月10万円稼いだ事例を読みたい」のような具体度が必要。

【響いた箇所】
- なし（原稿がない）

【書き直し提案】
- intake では「どんな読者が読んだら一番嬉しいか」を1人の像で答えてもらう。
